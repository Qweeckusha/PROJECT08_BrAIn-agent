# Документация: `query.py` (Ядро чтения)

Этот файл отвечает за **поиск по базе знаний**, **генерацию ответа** и **потоковую отдачу** результата. Работает полностью локально через LM Studio и FAISS.

## Библиотеки
langchain_community - расширяет функционал
langchain_core - основной функционал
langchain_openai - интеграция с OpenAI API моделями

## Архитектура пайплайна
```
Вопрос → Поиск в FAISS → Форматирование контекста → Промпт → LLM → Ответ (SSE)
```

---

## Загрузка векторной базы (FAISS)
```python
db = FAISS.load_local(FAISS_INDEX, embeddings, allow_dangerous_deserialization=True)
retriever = db.as_retriever(search_kwargs={"k": 5})
```
**Что делает:** Поднимает заранее созданный индекс FAISS с диска. `retriever` — это поисковик, который по тексту запроса находит `k` самых похожих чанков.


## Подключение локальной LLM (LM Studio)
```python
llm = ChatOpenAI(
    base_url="http://localhost:1234/v1",
    api_key="lm-studio",
    model="qwen/qwen3-4b-2507",
    temperature=0.5
)
```
**Что делает:** Обёртка над OpenAI-совместимым API. LM Studio поднимает локальный сервер на порту `1234`, а LangChain общается с ним как с облачной моделью.
**⚙️ Настройки:**
- `temperature=0.5` — баланс между точностью и вариативностью. Для RAG-систем лучше `0.1–0.3`, чтобы модель меньше "фантазировала". Но 0,5 на тестах показала себя довольно хорошо.
- `api_key="lm-studio"` — заглушка, локальный сервер её не валидирует (`api_key` является обязательным полем)


## Системный промпт
```python
prompt = ChatPromptTemplate.from_template("""...""")
```
**Что делает:** Задаёт "правила игры". Жёстко привязывает модель к контексту, запрещает выдумки и диктует структуру ответа.
**Ключевые директивы:**
- `Строго на контексте` - защита от галлюцинаций. Работает только с имеющимися в базе знаний записями
- `Связывай записи` - модель синтезирует информацию, а не копирует сухую теорию. Может спокойно рассказать своими словами и с примерами
- `Стоп-слова` - убирает роботизированные вводные (`"На основе предоставленных данных..."`). Вообще необязательно, но пусть будет.


## LCEL-цепочка (Chain)
```python
chain = (
    {
        "context": itemgetter("question") | retriever | RunnableLambda(format_docs),
        "question": itemgetter("question")
    }
    | prompt | llm | StrOutputParser()
)
```
**Что делает:** Собирает всё в один конвейер через LangChain Expression Language (LCEL).
**Поток данных:**
1. `itemgetter("question")` вытаскивает вопрос из входного словаря `{"question": "..."}`. Типа костыль
2. Отправляет в `retriever` → получает список `Document` → `RunnableLambda(format_docs)` превращает их в строку
3. Подставляет `{context}` и `{question}` в шаблон промпта.
4. Прогоняет через LLM → `StrOutputParser()` очищает вывод от лишних обёрток.



## Потоковая отдача (`stream_query`)
```python
def stream_query(question: str):
    ...
    yield f'event: sources\ndata: {json.dumps(sources)}\n\n'
    for chunk in chain.stream({"question": question}):
        yield f'event: answer\ndata: {chunk}\n\n'
```
**Что делает:** Генератор для Server-Sent Events (SSE). Идеально для фронтенда, чтобы показывать ответ "по словам" и сразу рисовать сайдбар с источниками.
**📦 Формат событий:**
| Event      | Payload                          | Назначение                          |
|------------|----------------------------------|-------------------------------------|
| `sources`  | JSON `[{topic, preview}, ...]`   | Мгновенная отрисовка источников     |
| `answer`   | Строка (чанк токена)             | Потоковый вывод ответа              |
| `done`     | `{}`                             | Сигнал завершения стрима            |
